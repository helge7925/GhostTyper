# Bug-Fixes 2025-02-17

## Bug 1: iPhone Datei-Upload öffnet Kamera statt Dateiverwaltung

**Status:** Behoben in v1.0.1

### Problem
Beim Klick auf den "Datei hochladen"-Bereich im Transkriptions-Panel wurde auf iOS-Geräten (iPhone/iPad) direkt die Kamera geöffnet, statt die Dateiverwaltung zu zeigen.

### Ursache
Das `<input type="file">` Element hatte das Attribut `capture="environment"` gesetzt:

```html
<!-- Alt (Problem): -->
<input type="file" accept="audio/*" capture="environment" />
```

Das `capture` Attribut ist laut HTML-Spezifikation dafür gedacht, direkt eine Kamera/Mikrofon-Aufnahme zu starten, anstatt eine Dateiauswahl zu ermöglichen.

### Lösung
Das `capture="environment"` Attribut wurde entfernt:

```html
<!-- Neu (Fix): -->
<input type="file" accept="audio/*" />
```

### Betroffene Datei
- `components/AudioUploadForm.js` (Zeile 148)

### Referenz
- [MDN: HTML attribute: capture](https://developer.mozilla.org/en-US/docs/Web/HTML/Attributes/capture)

## Bug 2: Bulletpoints können nicht eingerückt/ausgerückt werden

**Status:** Behoben in v1.0.1

### Problem
Im Editor konnten Bulletpoints (Listenpunkte) nicht eingerückt oder ausgerückt werden. Die Tab-Taste funktioniert nicht in `contentEditable` Bereichen, und es gab keine UI-Elemente für diese Aktion.

### Ursache
Die Editor-Toolbar bot keine Buttons für `indent` und `outdent` Befehle.

### Lösung
Zwei neue Buttons zur Formatierungsleiste hinzugefügt:
- Einrücken-Button (rechter Pfeil ins Zeichen)
- Ausrücken-Button (linker Pfeil aus Zeichen)

Beide nutzen die Browser-native `document.execCommand()` API:
- `indent` – Rückt die aktuelle Zeile/Listenebene ein
- `outdent` – Rückt die aktuelle Zeile/Listenebene aus

### Betroffene Datei
- `components/DocumentEditor.js` (Toolbar-Bereich, nach der Listen-Button-Gruppe)

### UI-Änderung
```
[... bestehende Formatierungs-Buttons ...]
[Liste] [Nummerierte Liste] | [Einrücken →] [← Ausrücken]
```

### Referenz
- [MDN: Document: execCommand()](https://developer.mozilla.org/en-US/docs/Web/API/Document/execCommand)

## Bug 3: PDF-Export fällt ständig in Fallback-Modus

**Status:** Behoben in v1.0.1

### Problem
Der PDF-Export fiel regelmäßig in den Fallback-Modus (Browser-Vorschau), weil der Chromium-Pfad nicht gefunden wurde oder falsche Pfade geprüft wurden.

### Ursache
1. Auf macOS wurden die typischen Installationspfade nicht berücksichtigt
2. Fehlende detaillierte Logging-Informationen zur Diagnose
3. `PDF_CHROMIUM_PATH` wurde nicht korrekt validiert

### Lösung
1. Plattformspezifische Pfade hinzugefügt:
   - macOS: `/Applications/Google Chrome.app`, Homebrew-Pfade
   - Linux: Zusätzliche Chromium-Pfade
2. Verbesserte Logging-Ausgaben für bessere Diagnose
3. Bessere Validierung von `PDF_CHROMIUM_PATH`

### Betroffene Datei
- `lib/pdf-export.js` (`resolveChromiumPath` Funktion)

### Workaround für Entwickler
Falls Chromium weiterhin nicht gefunden wird:
```bash
# macOS mit Chrome
export PDF_CHROMIUM_PATH="/Applications/Google Chrome.app/Contents/MacOS/Google Chrome"

# Linux
export PDF_CHROMIUM_PATH="/usr/bin/chromium"
```

## Bug 4: Word-Export quetscht Listenstruktur in Fließtext

**Status:** Behoben in v1.0.1

### Problem
Beim Word-Export (DOCX) wurden:
- Listenstrukturen zu Fließtext
- Formatierungen (fett, kursiv) verloren
- Verschachtelte Listen nicht dargestellt

### Ursache
Der DOCX-Parser in `export-utils.js` war zu simpel und hat nur `textContent` verwendet:
```javascript
// Alt (Problem):
children.push(createBrandRun(node.textContent));
```

### Lösung
HTML-Parser komplett überarbeitet:
- Rekursive Verarbeitung von HTML-Knoten
- Korrekte Behandlung von `<strong>`, `<em>`, `<u>` Tags
- Verschachtelte Listen (`<ul>`, `<ol>`, `<li>`) werden korrekt als `docx`-Listen exportiert
- Formatierungen werden als `TextRun`-Optionen übertragen

### Betroffene Datei
- `lib/export-utils.js` (`exportToDoc` und neue `parseHtmlNode` Funktion)

### Technische Details
Neue `parseHtmlNode()` Funktion rekursiv durch DOM:
- Text-Nodes → `TextRun`
- `<h1>`-`<h3>` → `Paragraph` mit Überschriftenformatierung
- `<ul>`/`<ol>` → `Paragraph` mit Bullet/Numbering
- `<strong>`/`<b>` → `TextRun` mit `bold: true`
- `<em>`/`<i>` → `TextRun` mit `italics: true`
- `<u>` → `TextRun` mit `underline`
- Verschachtelte Strukturen werden korrekt aufgelöst
